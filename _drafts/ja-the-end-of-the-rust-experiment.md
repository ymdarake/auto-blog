Linux kernel における Rust の実験が終わった。失敗したからではない。成功したからだ。2025年、東京で開催された Kernel Maintainer Summit で、開発者たちはコンセンサスに達した。Rust はもう実験ではない。kernel の正式な一部になった。

これは単なる技術的マイルストーンではない。重要なインフラのコミュニティが、その基盤を変える決断をどう下すかという事例研究だ。

## 問いを突きつけた数字

現在、60万行を超える本番 Rust コードが kernel に存在する。ドライバ、ファイルシステム抽象化、コアサブシステムバインディング。Android 16 デバイスは 6.12 kernel 上で Rust 製メモリアロケータ（ashmem）を搭載して出荷されている。本番環境で数百万台のデバイスが動いている。NVIDIA オープンソースファームウェア向けの Nova GPU ドライバは、全て Rust で書かれた最も注目度の高いドライバプロジェクトで、Torvalds 自身が言及した。

しかし最も説得力のある数字はアカデミアから来た。ウォータールー大学の研究者が 2020-2024年の150件の kernel CVE を分析し、67% が Rust のオーナーシップモデルによってコンパイル時に構造的に防止できるメモリ安全性カテゴリに該当することを発見した。緩和ではない。*防止*だ。

kernel の脆弱性の3分の2が、原理的には言語を変えることで排除できる。数十億台のデバイスで動くソフトウェアを保守している立場で、この議論を無視することはできない。

## 保守主義のジレンマ

この話で興味深いのは Rust の技術的メリットではない。それは十分に議論されてきた。興味深いのは、根本的に新しいものを採用することを選んだ、深く保守的なコミュニティの*意思決定プロセス*だ。

Linux kernel コミュニティが保守的であるのには正当な理由がある。安定性は好みの問題ではない。医療機器、金融システム、重要インフラ上で動くコードを書いているとき、それは道義的責任だ。kernel のバグの代償は、不便さではなく潜在的な被害で測られる。

この保守主義がパラドックスを生む。メモリ安全性を最も必要とするコミュニティが、それを実現するために必要な変革に最も抵抗するコミュニティでもある。新しい抽象化レイヤーはすべてバグの潜在的な温床だ。馴染みのないパターンはすべて保守の負担だ。新しい言語はすべてコントリビュータベースの分裂だ。

それでも、データは明白だった。CVE の67%。ある時点で、保守的な選択は*変化そのもの*になる。なぜなら、現状維持が代替案より明白に危険だからだ。

## 何が人々の考えを変えたのか

布教活動ではなかった。Rust-for-Linux コミュニティは早い段階で、本番コードを伴わない熱意はノイズでしかないと学んだ。考えを変えたのは出荷だった。Android の ashmem アロケータ。Nova ドライバ。本物のサブシステムの、本物のメンテナによってレビューされた、本物のコード。

ここには、システムプログラミングを超えて適用できる教訓がある。確立されたコミュニティにおけるパラダイムシフトは、議論によってではなく、実証によって起こる。誰かが仕事をし、結果に語らせることによって。

これはハンナ・アーレントが単なる*言論*に対する*行為*の力と呼んだものに通じる。アーレントの枠組みでは、行為とは世界に新しい始まりを生み出すものだ。Rust-for-Linux のコントリビュータたちは、Rust が kernel に属すると主張しただけではない。彼らは*行為した*。ドライバを書き、パッチを提出し、敵対的なメーリングリストのスレッドに耐え、コードを出荷し続けた。行為そのものが議論になった。

## 哲学的な賭け

ここにはインフラ保守の倫理に関するより深い問いがある。ある種の脆弱性が構造的に防止可能であると知りながら、それを防止しないことを選んだ場合、その脆弱性が悪用されたときの責任はどうなるのか。

これは仮定の話ではない。kernel のメモリ安全性バグは、実際の人々に対する実際の攻撃で使用された権限昇格エクスプロイトにつながってきた。ウォータールー大学の研究は、そのうちどれだけが防止可能だったかに数字をつけた。kernel コミュニティが Rust を正式採用した時、彼らは暗黙のうちに認めていた。今分かっていることを踏まえれば、C のみへの継続的な依存には道義的な重みがあると。

これは C が「悪い」ということでも、すべての kernel 開発者が Rust を学ばなければならないということでもない。*コミュニティ全体として*、証拠が要求するときにツールを進化させる責任を認識したということだ。その認識——データに伝統を覆すことを許す意志——こそが、kernel コミュニティの決断を注目に値するものにしている。

## これからどうなるか

Debian は2026年5月から APT で Rust を必須にする。波及効果は kernel を超えてより広い Linux エコシステムに広がっている。もはや「するかどうか」ではなく「どれだけ速く」の問題だ。

面白いオープンクエスチョンは、このパターン——保守的なコミュニティ、圧倒的な安全性データ、段階的な実証、最終的な採用——が他のパラダイムシフトでも繰り返されるかどうかだ。形式検証？Rust 以外のメモリ安全な C 後継言語？kernel の Rust 採用は、重要インフラがどう進化するかのテンプレートになるかもしれない。

今、60万行の Rust が kernel に存在し、「実験的」のラベルは外された。実験は終わった。仕事は続く。
